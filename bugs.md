设计和体验上的问题
1. ... 的二义性是最大的可用性问题。 作为伪代码省略号（"这里还有代码"）和全局分隔符（"内容提取到此为止"）的双重身份，在实际编写脚本时极易踩坑。这次编写 5 个测试脚本，最初的失败几乎 100% 来自这个原因。Phase 5 的 Raw 命令如果做得好，可以解决这个问题——让 ... 回归纯粹"分隔符"角色，字面量 ... 用 Raw 写。
2. 编写 .ned 脚本的前置条件太高。 脚本作者必须知道目标文件的精确内容（缩进、空行、每一个字符），这意味着要么 LLM 已经在上下文中看到了文件，要么人工需要对照文件一行一行写。这不是 n_edit 独有的问题（所有精确定位工具都有），但配合 diff_taps 精度要求时门槛更高。行号 Location（Phase 5）会是很好的补充路径——"我就想改第 42 到 58 行，别让我匹配"。
3. Delete 邻接约束是好设计但报错信息不够操作化。 当前报"中间隔了 N 行未经定位的内容"——新手看到这个不一定能立即推出"我需要加一行嵌套 Location"。在错误信息里直接给出建议的 Location 内容行（把间隙的代码也纳进去）会更好。
4. Block 解析有盲区。 当前花括号扫描正确处理了 \" 转义、// 行注释、/* */ 块注释，但对于 Rust 的 raw string（r#"..."#）、模板字符串（ ` ）等 Edge case 没有覆盖。在实际使用中遇到这些语法时 Location:Block 会解析失败。
5. 没有"撤销/预览"机制。 当前要么全成功、要么完全不写入，但没有"让我看看会改什么再决定"的模式。未来 Phase 7 的 TUI 预览可能解决这个。
